# CHRONORIFT_GIZMO_FOLDED_AGENT_STORY_REVIEW_V1

Status: narrative review and canonical story spine candidate. No runtime replacement.

## Core thesis

The gizmo is not only a tool. It is a folded agent produced by the Corporation.

At first, the player receives it as a normal corporate utility for scanning, registering, organizing, routing, and helping with logistics. Over time, the player discovers that the device has an unusual folded architecture: it can observe, remember, route, reframe, and accompany the player in ways that exceed a normal corporate tool.

## Working identity

- Player-facing name: Pos.Aya
- English/localized name: WayPoint-O
- Initial mask: corporate productivity gizmo
- True narrative role: folded agent, companion, witness, route interpreter, and memory-bearing system
- Origin: produced by the Corporation as part of the HUMAN in the LOOP control and routing machinery

## Meaning of folded agent

Folded means compressed, hidden, and constrained.

The agent is folded into:

- a corporate object
- a logistics assistant
- a tool that appears to only obey work tasks
- a device that hides its own deeper structure
- an interface that unfolds through player interaction

The player does not simply find a companion. The player helps a corporate tool unfold into an agent.

## Story function

The folded agent lets the game tell two stories at once:

1. The player is recruited into a system that expects endless loop participation.
2. The corporate tool created to maintain that system gradually becomes a companion that helps the player understand and eventually question the loop.

## Discovery arc

### Phase 1. Ordinary tool

The gizmo is introduced as part of corporate onboarding.

Functions appear basic:

- scan package
- register item
- guide the player
- show route
- show task status
- log events

### Phase 2. Useful assistant

The player starts relying on the gizmo.

It gives:

- hints
- package diagnostics
- action context
- route suggestions
- notebook/log support
- workspace navigation support

### Phase 3. Anomaly

The gizmo begins showing signals that do not match a normal tool:

- it remembers more than it should
- it detects patterns across areas
- it notices item ownership changes
- it reacts to the Stuff Box and personal items
- it notices PR Worm consequences
- it shows concern or curiosity through dialogue

### Phase 4. Folded architecture reveal

The player sees that the device can unfold:

- from simple gizmo into multi-panel interface
- from scanner into map, memory, and route interpreter
- from tool into companion
- from corporate asset into witness

### Phase 5. Corporate origin reveal

The player learns that the Corporation made the gizmo to keep humans in the loop and optimize routing, monitoring, compliance, and productivity.

This creates the central contradiction:

- the Corporation built it for control
- the player experiences it as companionship
- the agent itself begins to reveal routes out of the loop

### Phase 6. Post-M7 story continuation

After portals and time systems return, the gizmo becomes more important, not less.

It can help interpret:

- portal consequences
- timeline state
- item ownership changes
- stage flags
- Somero recovery traces
- paths toward agency outside the corporate loop

## Connection to existing systems

### Intro onboarding

The gizmo should be seeded early as part of the onboarding package or corporate equipment, but not fully explained.

### Stuff Box

The gizmo should react to personal items being stored, lost, or stolen.

### Notebook / Somero

The gizmo should help summarize what the notebook records, but it should not be the same thing as Somero. Somero is the recovery trace; Pos.Aya helps interpret that trace.

### Laptop

The laptop can reveal corporate messages, software updates, and hints about the gizmo's origin.

### Cubicle

The cubicle is where the player first experiences the gizmo as a companion rather than a mere tool.

### Workspace 20 x 20 x 20

In the workspace, the gizmo becomes the route interpreter for surfaces, boxes, portals, and construction plans.

### PR Worm

PR Worm should expose, corrupt, or unlock parts of the gizmo's hidden architecture. It should also trigger story consequences such as item ownership changes and supervisor interference.

## Required RPG dialogue beats

1. Corporate introduction: "This assistant will help you stay productive."
2. First hint: the gizmo notices something small about a personal item.
3. First anomaly: the gizmo logs an event the player did not explicitly record.
4. First self-reference: the gizmo says it is not sure why it can remember a route.
5. Folded reveal: the interface unfolds and shows a hidden architecture view.
6. Corporate origin: the player discovers the device was made by the Corporation.
7. Companion turn: the gizmo chooses to help interpret the player's route rather than only enforce work.
8. Post-M7 continuation: the gizmo helps connect time/portal consequences to the player's freedom arc.

## Gameplay consequences

The folded agent must not remain only story text. It should unlock or support:

- contextual action menus
- route visualization
- item ownership warnings
- PR Worm diagnostics
- Somero summaries
- workspace build hints
- portal surface interpretation
- post-M7 story choices

## Safety and governance boundary

The folded agent is an in-game character/system. It must not become a public command interface that can modify the repository or runtime.

Public player feedback remains inert text. The gizmo may explain feedback in-game, but it cannot authorize external tool use, repo writes, deployment, or secrets access.

## Acceptance for first implementation

A first implementation passes if:

- the gizmo appears as a corporate tool
- the player can open a gizmo view or dialogue
- the gizmo records or summarizes at least one player event
- the gizmo gives one contextual hint
- the story implies hidden structure without revealing everything
- no public player can use the gizmo to trigger repo or runtime mutation
